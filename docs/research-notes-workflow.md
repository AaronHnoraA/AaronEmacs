# Research Notes Workflow

这份文档回答一个实际问题：

- 科研笔记到底用 `org`
- 还是用 `jupytext` / `ipynb`
- 要不要让 `org` 再同步成 notebook

这套配置里，执行层已经统一到 Jupyter。`org` 里的 `python` / `sage` / `maple` 源块本身就会走 Jupyter 后端，`jupytext` 则是另一条脚本优先的 notebook 工作流。

所以这里不建议把它们再强行串成一条 `org -> jupytext -> ipynb` 的双向同步链。

## 1. 核心原则

不要把 `org` 和 `ipynb` 维护成同一份内容的两个真源。

更稳的做法是：

- `org` 是知识库、研究日志、结构化笔记
- `jupytext` / `ipynb` 是实验工作台、交互运行环境、分享载体
- 稳定代码放进 `src/`、脚本或包里，不长期埋在文档里

一句话：

- `org` 管“想法、问题、结论、结构”
- `jupytext` 管“试验、输出、图、交互”
- `src` 管“复用代码”

## 2. 为什么不建议 `org -> jupytext -> ipynb`

原因很简单：`org` 和 notebook 不是同一种文档模型。

`org` 强在：

- heading 层级
- TODO 和 agenda
- property drawer
- 跨主题组织
- 导出和长文档写作

notebook 强在：

- cell 执行顺序
- 输出和图像原地保存
- 交互式调试
- 给别人直接用 Jupyter 打开

如果你想让两边都当主格式，通常会出现几个问题：

- 为了迁就 notebook，`org` 会越写越不像 `org`
- 为了迁就 `org`，notebook 会失去自然的 cell 结构
- 元数据、标题结构、输出、折叠语义并不天然一一对应
- 最后你要处理的是同步问题，不是研究问题

所以更合理的思路不是“双向同步”，而是“按用途分工”。

## 3. 怎么分工

### 3.1 什么时候用 `org`

下面这些内容，优先放 `org`：

- 研究问题定义
- 文献摘录和批注
- 推导、证明、思路比较
- 实验计划
- 每日研究日志
- 对多个实验结果的对照总结
- 最终结论和下一步行动

如果一个文件的重点是“把事情讲清楚”，它就应该是 `org`。

### 3.2 什么时候用 `jupytext` / `ipynb`

下面这些内容，优先放 `jupytext` 或 notebook：

- 快速试一个想法
- 大量 cell 反复运行
- 强依赖输出、表格、图像
- 需要用 JupyterLab 调试
- 需要把 notebook 发给别人
- 需要保留可视化输出快照

如果一个文件的重点是“先跑出来看看”，它就应该是 notebook 工作流。

### 3.3 什么时候抽到 `src/`

下面这些内容，不应该长期留在 `org` 或 notebook：

- 重复使用的函数
- 数据清洗和特征工程逻辑
- 模型定义
- 公共绘图函数
- 实验配置读取
- 评估和导出流程

判断标准很简单：

- 第二次复制粘贴时，就该考虑抽走了

## 4. 推荐目录模型

你不必完全照抄，但建议至少分成三层：

```text
project/
  notes/
    inbox.org
    daily.org
    topic-a.org
    paper-reading.org
  experiments/
    exp-001-baseline.py
    exp-001-baseline.ipynb
    exp-002-ablation.py
    exp-002-ablation.ipynb
  src/
    data.py
    model.py
    plotting.py
  outputs/
    figures/
    tables/
  README.md
```

含义是：

- `notes/` 放结构化知识和研究叙事
- `experiments/` 放实验现场
- `src/` 放稳定逻辑
- `outputs/` 放导出的结果物

## 5. 日常工作流

这是比较稳的一套日常顺序。

### 5.1 新想法阶段

1. 在 `notes/inbox.org` 或当天的研究日志里写下问题。
2. 明确这次实验要回答什么，不要一上来就开跑。
3. 如果需要交互探索，在 `experiments/` 新建一个 `jupytext` 配对实验。

这一步的目标不是写完整，而是定义问题。

### 5.2 试验阶段

1. 在 `experiments/` 里跑 notebook 或 `jupytext` 脚本。
2. 看输出、调参数、画图、排查异常。
3. 一旦某段代码开始稳定，抽到 `src/`。
4. notebook 里只保留本次实验真正相关的编排。

这一步的目标是找答案，不是整理叙事。

### 5.3 沉淀阶段

实验一旦回答了一个具体问题，就回到 `org`，把下面几项整理进去：

- 问题是什么
- 数据或环境是什么
- 核心方法是什么
- 关键结果是什么
- 结果怎么解释
- 结论是什么
- 下一步是什么

这里可以只贴少量最关键的代码块，不要把整个实验原样塞回笔记。

### 5.4 收束阶段

当一个实验已经结束：

- 保留 `ipynb` 作为实验记录或分享件
- 保留配对脚本作为可版本控制的主实验文本
- 在 `org` 中保留解释、结论和索引

这里推荐的主线是：

- 实验真相在 `experiments/`
- 研究理解在 `notes/`

## 6. 单向流动，不做双向真源

推荐的数据流向是：

```text
问题/计划 -> org
探索/运行 -> jupytext or ipynb
稳定逻辑 -> src
结论/总结 -> org
分享/展示 -> ipynb or exported output
```

不要追求：

```text
org <-> jupytext <-> ipynb
```

因为这会把系统变成“同步系统”，而不是“研究系统”。

## 7. 你的这套 Emacs 配置里怎么落地

这套配置已经把两条路径都配好了：

- `org` 源块可以直接用 Jupyter 后端执行
- `jupytext-mode` 负责脚本和 notebook 配对同步

所以你的选择不应该是“二选一”，而应该是：

- 笔记层用 `org`
- 实验层用 `jupytext`
- 两边都用 Jupyter 当执行后端

这正好避免了重复造转换链。

## 8. 一个简单判断法

写任何新文件前，先问自己一句：

- 这个文件的核心是“讲清楚”还是“跑出来”？

如果答案是：

- “讲清楚”，用 `org`
- “跑出来”，用 `jupytext` / `ipynb`

再补一句：

- “以后还会复用”，抽到 `src/`

## 9. 最小规则集

如果你只想记住几条规则，就记这五条：

1. 同一份核心内容只保留一个真源。
2. `org` 写问题、过程、结论，不承担所有实验输出。
3. `jupytext` / notebook 写实验现场，不承担长期知识组织。
4. 重复逻辑尽快抽到 `src/`。
5. 从实验回流到笔记时，整理结论，不复制整份 notebook。

## 10. 推荐起步法

如果你现在还没完全定型，可以直接这样开始：

- 每个项目一个 `notes/` 目录
- 每个项目一个 `experiments/` 目录
- 每个项目一个 `src/` 目录
- 每次新实验先在 `notes` 里写一句“我要回答什么”
- 实验结束后，把结论写回 `notes`

这已经足够形成一个稳定系统。

后面你再决定要不要加：

- `daily.org`
- 文献阅读模板
- 实验编号规范
- 结果索引页

但这些都属于优化，不是前提。
