#+title: Jupyter-Centered Org and Jupytext Workflow

* Scope

This document describes the current Jupyter-centered workflow in this Emacs
config.  It covers:

- Org Babel execution through =ob-jupyter=
- Existing-kernel workflows based on =kernel-*.json= connection files
- Remote kernels, including =remote_ikernel=-style kernelspecs
- The script-first Jupytext workflow in =lisp/init-jupyter.el=
- How to add or maintain languages and kernels later

Relevant config entry points:

- =lisp/init-jupyter.el=
- =lisp/init-org-babel.el=

* Design

The current setup treats Jupyter as the unified execution backend.

From the user's point of view:

- Plain Org languages like =python=, =sage=, =sagemath=, and =maple= stay plain
- REPL, Org execution, async execution, and existing-kernel attachment all use
  the Jupyter protocol underneath

In practice the stack is:

- Org / source buffers
- =ob-jupyter= and =emacs-jupyter=
- Jupyter kernel protocol
- A concrete kernel for the target language

This means Jupyter acts as a compatibility layer for execution and session
management.  It does *not* invent language support.  A language must have a
working Jupyter kernel, or a running existing kernel with a readable connection
file, to participate in this workflow.

* Current Command Surface

The main keybindings live in =lisp/init-jupyter.el=.

- =C-c j r=: connect to the remembered existing kernel for the current language;
  if none is remembered, prompt for a =.json= connection file
- =C-u C-c j r=: start a new kernel through =jupyter-run-repl=
- =C-c j c=: explicitly choose a =.json= connection file and connect a REPL
- =C-c j ?=: open the full Jupyter dispatch menu
- =C-c j h=: open the Jupyter Hub management dashboard
- =C-c j C=: forget the remembered connection file for a language
- =C-c j k=: remember a connection file as the default existing kernel for a
  language
- =C-c j o=: update the current Org source block to use a chosen connection
  file and add =:async yes=
- =C-c j i=: inspect at point
- =C-c j e=: evaluate line or region through the connected REPL client
- =C-c j K=: choose the default kernelspec for a plain managed language such as
  =python= or =sage=
- =C-c j x=: refresh Jupyter kernelspec discovery and reconfigure Org/Jupyter
- =C-c j P=: install the current Python environment as a user kernelspec via
  =python -m ipykernel install --user=
- =C-c j y=: toggle =jupytext-mode=
- =C-c j s=: run =jupytext --sync= now
- =C-c j p=: register the current script buffer with its paired notebook

Org-specific async helpers live in =lisp/init-org-babel.el=.

- =C-c C-v C-a=: async execution helper for non-Jupyter sessionless blocks
- =C-c C-v C-k=: cancel the current async helper job

Jupyter-backed Org blocks use =ob-jupyter='s own async path and should normally
use =:async yes= directly.

* Jupyter Hub

The config includes a lightweight management dashboard for the whole Jupyter
stack:

- =C-c j h=

This opens =*Jupyter Hub*=, which shows:

- the current Org Babel language surface
- plain Org languages routed through Jupyter
- each language's effective default kernelspec
- remembered existing-kernel connection files
- all discovered kernelspecs with their language and display name

The Org Babel section makes the execution surface explicit:

- native loaded Babel languages such as =C=, =shell=, and =rust=
- plain languages such as =python= or =sage= that are routed through Jupyter
- generated =jupyter-*= aliases exposed by =ob-jupyter=

Inside the dashboard:

- =g=: redraw the dashboard
- =x=: refresh kernelspec discovery and redraw
- =K=: set the default kernelspec for the entry at point
- =d=: clear the default kernelspec override for the language at point
- =c=: remember a connection file for the language at point
- =C=: forget the remembered connection file for the language at point
- =r=: open a REPL for the language or kernelspec at point
- =P=: install the current Python environment as a kernelspec
- =D=: open the Jupyter doctor report
- =S=: prune remembered connection files that no longer exist
- =v=: open the local Jupyter runtime directory
- =l=: show the main =emacs-jupyter= log buffer
- =L=: show the kernelspec install log buffer
- =t=: show the Jupytext log buffer
- =?=: open the Jupyter dispatch menu
- =o=: open this workflow document
- =RET=: context action

The dashboard is intentionally simple: it is a display-and-manage tool, not a
replacement for raw =ob-jupyter= or direct connection-file workflows.

* Maintenance Surface

The config now includes a small maintenance layer in =lisp/init-jupyter-tools.el=
instead of continuing to grow =lisp/init-jupyter.el=.

Two entry points matter:

- =C-c j ?=: transient dispatch for the whole Jupyter surface
- =C-c j h=: persistent dashboard with inline actions

The maintenance commands cover:

- a =Jupyter Doctor= report for executables, runtime state, remembered
  connections, and current buffer Jupytext state
- pruning stale remembered connection files
- opening the local runtime directory directly
- jumping to the Jupyter, kernelspec-install, and Jupytext log buffers

* Org Babel Model

The config routes a small set of plain Org languages through Jupyter by default.

Current Jupyter-backed plain languages:

- =python=
- =sage=
- =sagemath=
- =maple=

This mapping is defined in =my/org-babel-jupyter-native-backends= in
=lisp/init-org-babel.el=.

Defaults for these languages are set so that:

- execution goes through =ob-jupyter=
- =:async yes= is the default
- if a remembered existing-kernel =.json= file exists for the language, that
  file becomes the default session
- if a user-selected default kernelspec exists for the language, it is preferred
  over the static built-in fallback kernelspec list
- otherwise the fallback symbolic Jupyter session name is used

** Plain block examples

Once a language is configured, plain blocks stay plain:

#+begin_src org
#+begin_src python :results output
print(2 + 3)
#+end_src
#+end_src

If =python= already has a remembered existing kernel, the block will run there.
Otherwise it falls back to the configured Jupyter session.

Explicit existing-kernel attachment:

#+begin_src org
#+begin_src python :session /Users/hc/Library/Jupyter/runtime/kernel-xxxx.json :async yes :results value
2 + 8
#+end_src
#+end_src

The same idea works for remote TRAMP connection files:

#+begin_src org
#+begin_src sage :session /ssh:your-host:/home/your-user/.local/share/jupyter/runtime/kernel-xxxx.json :async yes :results output
factorial(30)
#+end_src
#+end_src

** Existing kernel versus new kernel

These are not the same:

- =:session /path/to/kernel-xxxx.json=: attach to an already running kernel
- =:kernel some-kernelspec=: start a new kernel from that kernelspec

For example, this starts a new remote kernel if the kernelspec exists:

#+begin_src org
#+begin_src sage :kernel rik_ssh_aaron_wsl2_sage :results output
print(2 + 3)
#+end_src
#+end_src

This does *not* attach to an already running kernel.  For an existing WSL2 Sage
kernel, prefer a =.json= connection file instead.

* Environment-As-Kernel Workflow

This config now treats "Python environment" and "Jupyter kernelspec" as
separate but complementary layers.

- =direnv= controls which Python tooling Emacs sees in the current project
- Jupyter kernelspecs control which kernel =ob-jupyter= or =jupyter-run-repl=
  launches

This separation is intentional.  It means you do not need =conda.el= to make
project-local Python or notebook kernels work well in Emacs.

** Register the current environment as a kernel

If the current project environment already has =ipykernel= installed, use:

- =C-c j P=

This runs the equivalent of:

#+begin_src shell
python -m ipykernel install --user --name <kernel-name> --display-name <display-name>
#+end_src

from the current Emacs environment, which can be provided by =direnv=, =conda=,
=venv=, or another Python selector.

After that, use:

- =C-c j x=

to refresh kernelspec discovery inside Emacs.

** Make one kernelspec the default for a plain language

If you have multiple Python kernels, plain =#+begin_src python= blocks still
need a default kernelspec choice.  Use:

- =C-c j K=

Pick:

- a managed plain language such as =python= or =sage=
- a kernelspec name such as =python3=, =g6k=, or =my-project=

After that, plain blocks for that language prefer that kernelspec whenever they
are not pinned to an existing kernel connection file.

** What this buys you

- project-local editing can still be controlled by =direnv=
- one environment can be exposed as a reusable kernelspec
- =jupyter-run-repl= sees the environment by kernelspec name
- plain Org blocks can stay plain while still launching the right kernel

* Existing-Kernel Workflow

The primary workflow is now "connect to an already running kernel" rather than
"start a fresh kernel for every new session".

** Remember one existing kernel per language

Use:

- =C-c j k=

Pick:

- a language such as =python= or =sage=
- a =kernel-*.json= connection file

After that, plain Org blocks for that language default to the remembered
connection file, and =C-c j r= will connect a REPL to the remembered kernel.

Language alias sharing:

- =sage= and =sagemath= share the same remembered connection file

** Make one current Org block explicit

Use:

- =C-c j o=

This updates the current Org source block in place by adding:

- =:session /path/to/kernel-xxxx.json=
- =:async yes=

This is the easiest way to pin a specific block to a specific existing kernel.

** Local and remote connection files

Local:

- =/Users/hc/Library/Jupyter/runtime/kernel-xxxx.json=

Remote via TRAMP:

- =/ssh:host:/home/user/.local/share/jupyter/runtime/kernel-xxxx.json=

When the connection file is remote, =emacs-jupyter= can tunnel the remote kernel
ports over SSH.  This is the native remote existing-kernel path.

* Remote Kernels

There are three distinct remote stories worth keeping separate.

** 1. Existing remote kernel via connection file

This is the closest match to =nvim-ipy=-style usage.

Use:

- =jupyter-connect-repl=
- =C-c j c=
- =:session /ssh:host:/path/to/kernel-xxxx.json=

This attaches to a kernel that is already running remotely.

** 2. Start a new remote kernel on demand

This is the "remote kernelspec" or =remote_ikernel= story.

Examples:

- =rik_ssh_aaron_wsl2_sage=

Use this path when the remote kernelspec should *launch* a new remote kernel.
This is appropriate when you want remote startup convenience rather than
attachment to an existing runtime.

** 3. Jupyter server based remote sessions

=ob-jupyter= also has support for Jupyter server / TRAMP based sessions.  That
is a separate path from raw connection files.  For this config, raw connection
files are the preferred way to attach to an existing remote kernel because they
map directly onto the current REPL and Org workflow.

* Jupytext Workflow

The Jupytext workflow in =lisp/init-jupyter.el= is intentionally script-first.

The intended loop is:

1. Keep the notebook open and runnable in JupyterLab.
2. Save the notebook when you want outputs preserved.
3. Edit the paired script in Emacs.
4. Save the script and let =jupytext --sync= update the notebook.
5. In JupyterLab, reload the notebook from disk and continue.

** What the helper does

=jupytext-mode= is a buffer-local minor mode for the script buffer.

When enabled:

- the script is paired with a notebook
- save hooks run =jupytext --sync=
- the Emacs script buffer is reverted when the sync rewrites the file on disk

Useful commands:

- =C-c j y=: toggle =jupytext-mode=
- =C-c j p=: register the current script buffer and paired notebook
- =C-c j s=: sync immediately

The pairing logic stores:

- the script path
- the notebook path
- the Jupytext format, for example =py:percent= or =sage:percent=

** When to use Jupytext versus Org

Use Jupytext when:

- you want the notebook UI and outputs to remain first-class in JupyterLab
- you want to edit the paired script in Emacs
- you want notebook state preserved in notebook files

Use Org + Jupyter when:

- you want literate development inside Org
- you want plain source blocks with Jupyter-backed execution
- you want existing-kernel attachment and Org-native async results

These workflows are complementary.  They both rely on Jupyter, but they are not
the same editing model.

* Adding a New Language

Adding a new Jupyter-backed language is intentionally explicit.

Typical steps:

1. Ensure a working Jupyter kernelspec exists.
   For Python environments, =C-c j P= is the fastest path.
2. Add the language to =my/org-babel-jupyter-native-backends= in
   =lisp/init-org-babel.el=.
3. Add the language name to =my/jupyter-known-languages= in
   =lisp/init-jupyter.el=.
4. If the language has aliases that should share one remembered connection
   file, update:
   - =my/jupyter--normalize-language=
   - =my/jupyter--language-family=
5. If the language needs better current-buffer detection, update
   =my/jupyter-current-language=.

Minimal example for a hypothetical Julia setup:

#+begin_src emacs-lisp
("julia"
 :kernel-language "julia"
 :preferred-kernels ("julia-1.11")
 :session "julia"
 :mode julia
 :extension "jl")
#+end_src

The maintenance rule is simple:

- no kernel, no integration

If a language has no workable Jupyter kernel, it should not be routed through
this stack.

* Maintenance Notes

** State bug in emacs-jupyter

This config carries a local runtime fix for a native-compilation issue where
some =emacs-jupyter= monad helpers can fail with:

- =Symbol's value as variable is void: state=

The fix is applied in =lisp/init-jupyter.el= by rebinding the affected helper
functions after =jupyter-monads= loads.

** Connection files are runtime artifacts

Remembered connection files are useful, but they are tied to running kernels.

If a remembered =kernel-*.json= disappears or the kernel dies:

- reconnect to a new existing kernel
- re-run =C-c j k= for the language if needed
- or use =C-u C-c j r= to start a fresh kernel intentionally

** Remote sessions need the Jupyter transport stack

Remote attachment relies on Jupyter's normal transport machinery, including the
ZMQ path used by =emacs-jupyter=.  If remote sessions fail, verify:

- the remote kernel really exists
- the connection file still exists
- SSH / TRAMP access works
- the kernel ports are still live

** Verified behavior in this config

The current setup has been verified for:

- existing local kernel attachment through =kernel-*.json=
- Org =:results output=
- Org =:results value=
- Org =:async yes=
- REPL connection through remembered existing kernels
